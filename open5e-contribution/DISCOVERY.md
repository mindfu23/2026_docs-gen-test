# How a Docusaurus build surfaced a bug in a public API's OpenAPI spec

> The investigation that led to [`PR.md`](PR.md). A small, real example of docs tooling acting as
> a correctness check on the API it documents. (2026-06-24)

## Context

I was building **`2026_docs-gen-test`** — a hands-on comparison of five documentation/SDK tools
(Stainless, Mintlify, GitBook, Docusaurus, Promptless) run against **one shared OpenAPI spec**.
The unifying subject is the **Open5e** D&D 5e API (`https://api.open5e.com`), chosen because it
publishes a real, openly-licensed OpenAPI 3.0.3 spec — so the same spec could feed every tool.

For the comparison I work from three spec variants: the full spec (69 paths), a trimmed
**demo subset** (spells / creatures / classes), and an **auth** variant (subset + a fake
authenticated `POST`). The subset is what I pointed Docusaurus at.

## The failure

Scaffolded Docusaurus 3.10.1 + `docusaurus-plugin-openapi-docs`, then ran the reference generator:

```bash
npm run gen-api-docs
```

It errored:

```
Error: [Missing $ref pointer "#/components/schemas/Race". Token "Race" does not exist.]
 with footprint [components,schemas,SearchResult,properties,object,oneOf,4 ...]
WARNING: the following OpenAPI spec returned undefined: .../shared/openapi-subset.json
```

A `$ref` in the spec points at a schema component that doesn't exist — a **dangling reference**.
A `$ref` resolver (here, the Docusaurus OpenAPI plugin via Redocly) can't bundle the spec.

## Diagnosis

Walked the spec for refs whose target is undefined:

```python
# exactly one dangling ref, in both the full and subset specs:
# '#/components/schemas/Race'  — under  components/schemas/SearchResult/properties/object/oneOf
```

`SearchResult` is the response model for `/v2/search/` — its `object` field is a `oneOf` of every
searchable content type, and one of those options points at a `Race` schema that isn't defined.

## Is it me, or upstream?

Upstream — and **still live**. Re-fetched the production schema:

```bash
curl -s "https://api.open5e.com/schema/?format=json"
# Race defined in components: False
# refs to .../Race present:   1
# Species defined:            True   <-- the telltale
```

`Race` is gone; `Species` exists. That's the smell of a **rename that missed a spot**.

## Root cause

The Open5e spec is auto-generated by `drf-spectacular`, but the `SearchResult.object` `oneOf` is
**hand-built** in a post-processing hook (`server/oas.py`), with a candid comment:

```python
# Add a oneOf to the SearchResult object field. Couldn't figure out how to do this with @extend_schema_field
result['components']['schemas']['SearchResult']['properties']['object'] = {
    'oneOf': [
        { '$ref': '#/components/schemas/Item' },
        { '$ref': '#/components/schemas/Creature' },
        { '$ref': '#/components/schemas/Spell' },
        { '$ref': '#/components/schemas/CharacterClass' },
        { '$ref': '#/components/schemas/Race' }   # <-- stale; renamed to Species in v2
    ]
}
```

Open5e renamed **Race → Species** in v2 (issue **#704**, migration `0033_rename_race_species`),
but this hand-maintained `oneOf` wasn't updated. The fix is one line: `Race` → `Species`.

## Tracker check

Searched `open5e/open5e-api` issues and PRs — **not reported.** The closest is open issue **#950,
"Proposal: Commit the drf-spectacular generated OpenAPI spec in git."** This bug is a concrete
argument for it: had the generated spec been committed and `$ref`-validated in CI, the dangling
reference would have been caught at the rename.

## How I worked around it locally

So the comparison could proceed without waiting on upstream:
- **Subset / auth variants:** pruned the spec to only the components the kept paths use — which
  drops the unused `SearchResult` entirely, so the dangling `Race` ref goes with it.
- **Full variant:** removed the single dead `oneOf` entry in place.
All three variants now resolve with zero dangling refs.

## Takeaways

- **Docs tooling is a correctness check on the API.** A spec that "works" in a browser can still
  be invalid to a `$ref` resolver; running real codegen/doc-gen surfaces it.
- **Hand-patched generated artifacts drift.** The auto-generated parts of the spec followed the
  rename; the hand-written `oneOf` didn't. Anything hand-maintained alongside generated output
  needs a validation gate.
- **The cheap gate:** resolve every `$ref` (or bundle with Redocly) in CI. One assertion catches
  this whole class of bug — exactly what #950 enables.

→ The fix is drafted, ready to file, in [`PR.md`](PR.md).
